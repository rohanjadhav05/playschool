## Problem Statement

A local playschool and tuition centre in Karve, Karad, Maharashtra (Tier-2 city) has no online presence. The school has paying students and operates successfully on word of mouth, but parents who hear about it have no way to validate their decision digitally. The site needs to build trust through cultural alignment and activity proof — not generate leads from scratch.

The bigger idea: this website is v1 of a configurable template for Tier-2/3 Indian playschools. Atharva is the pilot. The goal is to deploy the same product for 10, 50, 100 similar schools across Maharashtra with minimal per-school effort.

---

## Demand Evidence

- School has paying enrolled students (operating, not idea stage)
- No purpose-built website template exists for Tier-2/3 Indian playschools — confirmed via search
  - Global templates (ThemeForest, Wix): English-only, feature-bloated, not India-specific
  - Enterprise solutions (Entab): target large K-12, not 20-30 student playschools
  - Custom dev shops: charge ₹20K–50K+ for one-off sites
- India edtech market projected at $29B by 2030, Tier-2/3 expansion is the growth story
- Zero competitors serving the Maharashtra playschool segment with a localized, WhatsApp-first, bilingual product

**Gap in evidence:** Parent behavior has not been directly observed. The trust-first hypothesis is reasonable but unvalidated. The Assignment (below) addresses this.

## Status Quo

When parents in Karad hear about a playschool, they currently:
- Ask neighbors/relatives who have kids (word of mouth dominates)
- Search on Google — land on nothing, or a Facebook page with irregular posts
- WhatsApp the teacher directly for fee/timing info
- Visit in person to make a judgment call

The website's job is to succeed the WOM step — validate the decision a parent has already started making, not generate cold intent. It must answer: "Is this school real? Does it share my values? Are these teachers good?"

---

## Target User & Narrowest Wedge

**User:** Indian parent, 25–40 years old, in a Tier-2/3 Maharashtra city. Primarily on an Android phone. Hindi or Marathi-medium comfort zone. Values cultural alignment (festivals, discipline, Indian food habits) as much as academic outcomes when selecting a playschool for a 3–5 year old.

**Narrowest wedge (stated by user):** Show proof of activities and cultural values. Not a form, not SEO — trust-building through activity photos, testimonials, and cultural alignment signals.

**Multi-school template wedge:** If deploying school #2 takes more than 30 minutes, the template model breaks. The config layer is the wedge for the business.

## Constraints

- Static frontend only — no backend, no SSR, no API calls (except S3 media)
- React + Vite + Tailwind + Framer Motion (as specced in PRD)
- S3 for all media assets
- Must work on low-end Android devices on 4G/3G
- No CMS — content configured via `school.config.js` per deployment
- Bilingual EN/Marathi via a `LanguageContext` React context exposing `{ lang, setLang, t(key) }`, backed by flat JSON objects (`en.json`, `mr.json`). No i18next. Falls back to `en` key if Marathi key is missing. Persists in `localStorage`.
- Lighthouse mobile score ≥ 85
- Deployment: Netlify or Vercel (free tier)

## Recommended Approach

**Approach B — Full PRD + Config Layer.**

Build the complete PRD spec exactly as written. On top of it, add a `school.config.js` at the project root that externalizes:
- School name, tagline, phone, WhatsApp, address
- Teacher name, experience, photo URL
- Year founded
- S3 bucket base URL
- Google Maps embed URL
- Social links
- Admission year, timing, batch size
- All per-school content that would differ between Atharva and school #2

The rest of the PRD components, design system, and file structure remain as-is. Every new school deployment is:
1. Fork the repo
2. Update `school.config.js`
3. Upload photos to their S3 bucket
4. Push to Netlify/Vercel

The bilingual toggle ships on Atharva and is instrumented with a simple localStorage counter to track toggle usage. Review at 90 days.

## Performance Requirements

Lighthouse mobile ≥ 85 requires explicit implementation:
- All below-fold images: `loading="lazy"` + `srcSet` (400w, 800w, 1200w WebP)
- Hero image: `<link rel="preload">` in index.html
- S3 assets must be served via CloudFront CDN (or Netlify's built-in CDN via Netlify Large Media) — raw S3 from ap-south-1 will be slow on 3G in Maharashtra
- Framer Motion: import only used components (`import { motion } from "framer-motion"`)
- All video `src` attributes loaded lazily via `IntersectionObserver`
- `font-display: swap` on Google Fonts
- Vite tree-shaking handles bundle size automatically on production build
